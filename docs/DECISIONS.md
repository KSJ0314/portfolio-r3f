## 기록

### 001. 브랜치 전략: 풀 Git Flow 채택

- **날짜**: 2026-07-01
- **상태**: 채택
- **맥락**: `main` 머지가 자동 배포로 이어지므로, 미완성·부분 기능이 라이브 사용자에게 노출될 위험이 있다. 운영 시작 시점은 예측 불가하고, 운영 중에도 변경이 계속된다.
- **결정**: `main + develop + feature`(+ 운영 시 `release`·`hotfix`) 구조의 Git Flow 채택. `develop`을 통합·스테이징 버퍼로 사용. 개발 단계에서는 `feature → develop → main` 직접 승격, `release`·`hotfix`는 운영 시작 후 도입.
- **이유**: `develop`이 라이브(`main`)와 개발을 분리해, 릴리스된 것만 프로덕션에 노출되게 한다. 처음부터 깔아두면 운영 전환 시 구조 변경이 불필요하다.
- **영향**: 초기 셋업은 `develop`에 모여 `main`으로 승격. Vercel 스테이징(`develop`)에서 검증 후 프로덕션(`main`) 반영.

### 002. 배포/호스팅: GitHub Pages → Vercel

- **날짜**: 2026-07-01
- **상태**: 채택 (기존 GitHub Pages 결정을 대체)
- **맥락**: 릴리스를 라이브 반영 전에 배포 환경에서 검증할 스테이징/프리뷰가 필요하다. GitHub Pages는 레포당 사이트가 1개라 스테이징·프로덕션 분리와 PR 프리뷰를 기본 기능으로 제공하지 못한다.
- **결정**: 배포처를 **Vercel**로 변경.
- **이유**: 브랜치·PR별 자동 프리뷰 URL 제공(`main`=프로덕션, `develop`=스테이징, PR=프리뷰). Git 연동 자동 CI/CD라 별도 워크플로가 불필요하고, Vite/React DX가 우수하며, base 경로·SPA 문제가 해소된다.
- **영향**: 로드맵 Phase 0.5를 "GitHub Pages + Actions" → "Vercel 연결"로 변경. 배포 워크플로 직접 작성 불필요.

### 003. 카메라 이동 모델: 캐릭터 팔로우 + 우클릭 홀드 이동

- **날짜**: 2026-07-07
- **상태**: 채택
- **맥락**: 캐릭터 없는 항공뷰에서 이동을 어떻게 표현할지. 초기엔 "타겟 글라이드"(카메라가 클릭 지점을 화면 중앙으로) 고려.
- **결정**: 임시 캐릭터를 두고 카메라가 캐릭터를 따라가 화면 중앙 고정. 우클릭 홀드로 이동, 좌클릭은 인터랙션용 예약.
- **이유**: 화면상 캐릭터 고정 + 월드 스크롤이 동물의숲/춘식이 톤에 맞고, Phase 13 실제 캐릭터·걷기 애니메이션으로 자연 확장. 우클릭/좌클릭 분리로 이후 클릭 상호작용과 충돌 없음.
- **영향**: 이동은 고정 속도 per-frame step, gsap 단발 트윈 제거.

### 004. Orthographic 뷰 채택 (원근 제거)

- **날짜**: 2026-07-07
- **상태**: 채택
- **맥락**: Perspective(fov)는 원근감이 있어 "춘식이의 관찰일기"식 평면 아이소메트릭과 다름.
- **결정**: Orthographic 카메라 채택(fov 없이 zoom으로 배율).
- **이유**: 원근 왜곡 없는 균일한 아이소메트릭 뷰, 위치 무관 동일 크기로 가독성↑.
- **영향**: 배율은 zoom(현재 120), 전환 시 이전 perspective 배율에 맞춤.

### 005. 이동 경계: clamp(투명 벽) + 최종 "바닥 전체 + 투명 경계벽"

- **날짜**: 2026-07-07
- **상태**: 채택
- **맥락**: 제한된 영역만 이동하되 유저에게 맵 가장자리를 안 보이게.
- **결정**: 목표점을 CAMERA_BOUNDS로 clamp해 경계에서 정지(투명 벽). 현재는 맵을 크게 둬 가장자리 숨김, 최종은 바닥 전체 + 투명 경계벽.
- **이유**: 안정적 탐험 UX, 콘텐츠를 항상 화면에 잡음.
- **영향**: 텍스처는 바닥 전체에 적용.

### 006. 스테이션 표현: 월드 내 고유 오브젝트 + 제자리 활성화 (Html-transform 벽 폐기)

- **날짜**: 2026-07-09
- **상태**: 채택 (기존 "2D 패널 = 월드에 뜬 벽" 모델 대체)
- **맥락**: 초기엔 모든 스테이션을 동일한 "월드에 뜬 벽"으로 두고, 인터랙션 시 카메라가 벽 정면으로 이동해 drei `<Html transform>`으로 2D DOM 패널을 띄우는 균일 모델을 가정했다.
- **결정**: 스테이션을 **각기 다른 고유 3D 오브젝트**로 만든다. 오브젝트 자체가 내용을 간략히 암시하는 미리보기이고, 좌클릭하면 **그 오브젝트가 제자리에서 활성화·확장**되어 상세를 보여준다. 상세 표현은 2D 패널로 한정하지 않고 **3D/혼합**도 허용한다. 공통층은 배치·근접·선택·미니맵·표지판뿐이고, 겉모습과 상세는 `스테이션 id → 전용 컴포넌트` 레지스트리로 스테이션마다 독립 구현한다.
- **이유**: 저폴리/아기자기 오픈월드 톤엔 균일한 벽보다 주제를 상징하는 오브젝트가 어울리고, 스테이션마다 표현·인터랙션이 완전히 달라도 서로 독립적으로 만들 수 있다. drei `<Html transform>`은 폐기가 아니라 상세가 2D일 때 쓰는 여러 옵션 중 하나로 격하된다.
- **영향**: PLAN의 "시그니처 경험 / 2D 패널=벽 / 프로젝트 전시 벽", 로드맵 Phase 3·4 문구를 오브젝트·활성화 모델로 갱신. Phase 2(배치)는 영향 적음 — 임시 오브젝트를 레지스트리에 등록해 프레임워크만 완성.

### 007. 스테이션 활성화: 라이프사이클만 공통층, 상세·카메라 연출은 스테이션별 전면 위임

- **날짜**: 2026-07-13
- **상태**: 채택
- **맥락**: 006에서 "스테이션마다 고유 오브젝트 + 제자리 활성화"로 정했지만, 활성화 이후의 상세와 카메라 연출을 공통 셸로 만들지 스테이션별로 만들지는 미정이었다. 공통 패널 셸 + 선언형 카메라 스펙(offset·zoom·duration)을 두는 안도 검토했다.
- **결정**: 공통층은 **활성화 라이프사이클**(`idle → entering → active → exiting`)·이동 잠금·마운트 자리·레지스트리 계약만 갖는다. 상세 UI와 카메라 연출은 전부 스테이션 구현의 몫이며 **공통 셸도 기본 구현체도 두지 않는다**. 활성화되면 공통층은 카메라 팔로우를 멈춰 **제어권을 스테이션에 넘기고**, 어떻게 움직이는지는 알지 못한다. 대신 닫힐 때 원래 오프셋·zoom으로 돌아오는 **복귀만 보장**한다.
- **이유**: 기본 구현체가 있으면 이후 스테이션들이 그 틀에 끌려간다. 카메라 연출도 스테이션마다 완전히 달라야 하므로 값 기반 선언형 계약조차 두지 않고, 트윈을 통째로 스테이션에 맡긴다.
- **상호작용 규칙**: 이동 잠금은 **애니메이션 재생 중(entering·exiting)에만** 걸린다. `active`에서 우클릭은 곧 종료 트리거이고, 클릭 지점을 기억해뒀다가 **종료 애니메이션이 끝난 뒤** 캐릭터가 그리로 출발한다. ESC로도 종료된다. 좌클릭은 상세 내부 요소 상호작용 전용이라 닫기에 쓰지 않는다.
- **계약**: 스테이션 구현은 `phase`를 받아 진입·종료 애니메이션을 재생하고, 끝나면 `enterComplete()`/`exitComplete()`로 공통층에 알린다. 등록된 구현이 없는 스테이션은 알릴 주체가 없으므로 공통층이 즉시 완료 처리한다.
- **한계(의도적으로 남김)**: 평상시 겉모습은 아직 `Station`의 공통 임시 박스이고, 활성화 시 마운트되는 상세 컴포넌트와 분리돼 있다. 그래서 같은 오브젝트가 이어서 변형되는 연출은 아직 만들 수 없다. 겉모습 슬롯은 고유 오브젝트를 만드는 Phase 8에서 레지스트리에 추가한다.
