## 기록

### 001. 브랜치 전략: 풀 Git Flow 채택

- **날짜**: 2026-07-01
- **상태**: 채택
- **맥락**: `main` 머지가 자동 배포로 이어지므로, 미완성·부분 기능이 라이브 사용자에게 노출될 위험이 있다. 운영 시작 시점은 예측 불가하고, 운영 중에도 변경이 계속된다.
- **결정**: `main + develop + feature`(+ 운영 시 `release`·`hotfix`) 구조의 Git Flow 채택. `develop`을 통합·스테이징 버퍼로 사용. 개발 단계에서는 `feature → develop → main` 직접 승격, `release`·`hotfix`는 운영 시작 후 도입.
- **이유**: `develop`이 라이브(`main`)와 개발을 분리해, 릴리스된 것만 프로덕션에 노출되게 한다. 처음부터 깔아두면 운영 전환 시 구조 변경이 불필요하다.
- **영향**: 초기 셋업은 `develop`에 모여 `main`으로 승격. Vercel 스테이징(`develop`)에서 검증 후 프로덕션(`main`) 반영.

### 002. 배포/호스팅: GitHub Pages → Vercel

- **날짜**: 2026-07-01
- **상태**: 채택 (기존 GitHub Pages 결정을 대체)
- **맥락**: 릴리스를 라이브 반영 전에 배포 환경에서 검증할 스테이징/프리뷰가 필요하다. GitHub Pages는 레포당 사이트가 1개라 스테이징·프로덕션 분리와 PR 프리뷰를 기본 기능으로 제공하지 못한다.
- **결정**: 배포처를 **Vercel**로 변경.
- **이유**: 브랜치·PR별 자동 프리뷰 URL 제공(`main`=프로덕션, `develop`=스테이징, PR=프리뷰). Git 연동 자동 CI/CD라 별도 워크플로가 불필요하고, Vite/React DX가 우수하며, base 경로·SPA 문제가 해소된다.
- **영향**: 로드맵 Phase 0.5를 "GitHub Pages + Actions" → "Vercel 연결"로 변경. 배포 워크플로 직접 작성 불필요.

### 003. 카메라 이동 모델: 캐릭터 팔로우 + 우클릭 홀드 이동

- **날짜**: 2026-07-07
- **상태**: 채택
- **맥락**: 캐릭터 없는 항공뷰에서 이동을 어떻게 표현할지. 초기엔 "타겟 글라이드"(카메라가 클릭 지점을 화면 중앙으로) 고려.
- **결정**: 임시 캐릭터를 두고 카메라가 캐릭터를 따라가 화면 중앙 고정. 우클릭 홀드로 이동, 좌클릭은 인터랙션용 예약.
- **이유**: 화면상 캐릭터 고정 + 월드 스크롤이 동물의숲/춘식이 톤에 맞고, Phase 11 실제 캐릭터·걷기 애니메이션으로 자연 확장. 우클릭/좌클릭 분리로 이후 클릭 상호작용과 충돌 없음.
- **영향**: 이동은 고정 속도 per-frame step, gsap 단발 트윈 제거.

### 004. Orthographic 뷰 채택 (원근 제거)

- **날짜**: 2026-07-07
- **상태**: 채택
- **맥락**: Perspective(fov)는 원근감이 있어 "춘식이의 관찰일기"식 평면 아이소메트릭과 다름.
- **결정**: Orthographic 카메라 채택(fov 없이 zoom으로 배율).
- **이유**: 원근 왜곡 없는 균일한 아이소메트릭 뷰, 위치 무관 동일 크기로 가독성↑.
- **영향**: 배율은 zoom(현재 120), 전환 시 이전 perspective 배율에 맞춤.

### 005. 이동 경계: clamp(투명 벽) + 최종 "바닥 전체 + 투명 경계벽"

- **날짜**: 2026-07-07
- **상태**: 채택
- **맥락**: 제한된 영역만 이동하되 유저에게 맵 가장자리를 안 보이게.
- **결정**: 목표점을 CAMERA_BOUNDS로 clamp해 경계에서 정지(투명 벽). 현재는 맵을 크게 둬 가장자리 숨김, 최종은 바닥 전체 + 투명 경계벽.
- **이유**: 안정적 탐험 UX, 콘텐츠를 항상 화면에 잡음.
- **영향**: 텍스처는 바닥 전체에 적용.
- **갱신(2026-07-15)**: clamp 방식이 곧 최종이다. 바닥을 이동범위보다 크게 둬 가장자리가 안 보이므로 별도 물리 경계벽은 두지 않는다. 제목의 "최종 바닥 전체 + 투명 경계벽"은 이 clamp 구현을 가리키며, 추가 작업은 없다.

### 006. 스테이션 표현: 월드 내 고유 오브젝트 + 제자리 활성화 (Html-transform 벽 폐기)

- **날짜**: 2026-07-09
- **상태**: 채택 (기존 "2D 패널 = 월드에 뜬 벽" 모델 대체)
- **맥락**: 초기엔 모든 스테이션을 동일한 "월드에 뜬 벽"으로 두고, 인터랙션 시 카메라가 벽 정면으로 이동해 drei `<Html transform>`으로 2D DOM 패널을 띄우는 균일 모델을 가정했다.
- **결정**: 스테이션을 **각기 다른 고유 3D 오브젝트**로 만든다. 오브젝트 자체가 내용을 간략히 암시하는 미리보기이고, 좌클릭하면 **그 오브젝트가 제자리에서 활성화·확장**되어 상세를 보여준다. 상세 표현은 2D 패널로 한정하지 않고 **3D/혼합**도 허용한다. 공통층은 배치·근접·선택·미니맵·표지판뿐이고, 겉모습과 상세는 `스테이션 id → 전용 컴포넌트` 레지스트리로 스테이션마다 독립 구현한다.
- **이유**: 저폴리/아기자기 오픈월드 톤엔 균일한 벽보다 주제를 상징하는 오브젝트가 어울리고, 스테이션마다 표현·인터랙션이 완전히 달라도 서로 독립적으로 만들 수 있다. drei `<Html transform>`은 폐기가 아니라 상세가 2D일 때 쓰는 여러 옵션 중 하나로 격하된다.
- **영향**: PLAN의 "시그니처 경험 / 2D 패널=벽 / 프로젝트 전시 벽", 로드맵 Phase 3·4 문구를 오브젝트·활성화 모델로 갱신. Phase 2(배치)는 영향 적음 — 임시 오브젝트를 레지스트리에 등록해 프레임워크만 완성.
- **갱신(2026-07-15)**: 공통층 책임에서 표지판을 뺀다. 방향 안내(표지판·바닥 화살표)는 스테이션 실물이 생기는 Phase 8에서 스테이션별로 넣는다. 미사용이던 `SIGNPOSTS`·`Signpost`·`HUB`도 제거했다.

### 007. 스테이션 활성화: 라이프사이클만 공통층, 상세·카메라 연출은 스테이션별 전면 위임

- **날짜**: 2026-07-13
- **상태**: 채택
- **맥락**: 006에서 "스테이션마다 고유 오브젝트 + 제자리 활성화"로 정했지만, 활성화 이후의 상세와 카메라 연출을 공통 셸로 만들지 스테이션별로 만들지는 미정이었다. 공통 패널 셸 + 선언형 카메라 스펙(offset·zoom·duration)을 두는 안도 검토했다.
- **결정**: 공통층은 **활성화 라이프사이클**(`idle → entering → active → exiting`)·이동 잠금·마운트 자리·레지스트리 계약만 갖는다. 상세 UI와 카메라 연출은 전부 스테이션 구현의 몫이며 **공통 셸도 기본 구현체도 두지 않는다**. 활성화되면 공통층은 카메라 팔로우를 멈춰 **제어권을 스테이션에 넘기고**, 어떻게 움직이는지는 알지 못한다. 대신 닫힐 때 원래 오프셋·zoom으로 돌아오는 **복귀만 보장**한다.
- **이유**: 기본 구현체가 있으면 이후 스테이션들이 그 틀에 끌려간다. 카메라 연출도 스테이션마다 완전히 달라야 하므로 값 기반 선언형 계약조차 두지 않고, 트윈을 통째로 스테이션에 맡긴다.
- **상호작용 규칙**: 이동 잠금은 **애니메이션 재생 중(entering·exiting)에만** 걸린다. `active`에서 우클릭은 곧 종료 트리거이고, 클릭 지점을 기억해뒀다가 **종료 애니메이션이 끝난 뒤** 캐릭터가 그리로 출발한다. ESC로도 종료된다. 좌클릭은 상세 내부 요소 상호작용 전용이라 닫기에 쓰지 않는다.
- **계약**: 스테이션 구현은 `phase`를 받아 진입·종료 애니메이션을 재생하고, 끝나면 `enterComplete()`/`exitComplete()`로 공통층에 알린다. 등록된 구현이 없는 스테이션은 알릴 주체가 없으므로 공통층이 즉시 완료 처리한다.
- **한계(의도적으로 남김)**: 평상시 모습은 아직 `Station`의 공통 임시 박스이고, 활성화 시 마운트되는 상세 컴포넌트와 분리돼 있다. 그래서 같은 오브젝트가 이어서 변형되는 연출은 아직 만들 수 없다. 비활성 슬롯은 고유 오브젝트를 만드는 Phase 8에서 레지스트리에 추가한다.
- **갱신(2026-07-22) — 상호작용 규칙 교체**: 우클릭 종료를 폐기한다. 닫는 방법은 **근접 이탈 · 스테이션이 제공하는 나가기 요소 · ESC** 세 가지이고, 우클릭은 순수하게 이동이다. 활성 중에도 평소처럼 걸어다닐 수 있으며 **영역에서 멀어지는 것이 곧 닫기**다. 이동 잠금은 `entering`에만 건다 — 걸어나가는 것이 종료 트리거인데 `exiting`에 잠그면 그 자리에서 멈칫한다. 종료 후 이동을 위한 `pendingTarget`은 쓸 일이 없어져 제거했다.
- **갱신(2026-07-22) — 한계 해소**: 레지스트리에 `Inactive`(비활성 상태, 상시 마운트) 슬롯을 추가해 비활성 모습도 스테이션이 직접 그린다. 근접 거리 계산을 맡기는 `distanceTo`도 함께 추가했다(DECISIONS 014). 스테이션이 부드럽게 복귀하려면 현재 카메라 자세를 보지 말고 `useCameraStore.followOffset`으로 복귀 자세를 계산해야 한다.

### 008. 아트 디렉션 전환: 저폴리 파스텔 마을 → 스케치북

- **날짜**: 2026-07-14
- **상태**: 채택 (기존 "저폴리 + 파스텔, 아늑한 섬/마을" 방향 대체)
- **맥락**: 기존 아트 방향은 동물의 숲·춘식이 톤의 아기자기한 마을이었다. 그러려면 나무·집·소품 등 배경 에셋을 대량으로 소싱하거나 모델링해야 하고, 에셋 팩을 쓰면 룩이 남의 것이 된다.
- **결정**: 월드를 **빈 스케치북**으로 둔다. 바닥은 손그림 모눈종이이고, 그 위에 필요한 요소만 그린다. 마을·나무·집·소품·하늘은 두지 않는다. **스테이션은 종이 위의 그림이고, 활성화하면 3D로 전환된다.** 전환 연출은 스테이션마다 다르게 만든다(2D 유지 + 카메라 회전 / 2D→3D 애니메이션 / 3D 오브젝트가 펼쳐짐 등).
- **이유**: "그림 → 3D 전환"이 이 포트폴리오의 시그니처이고, 배경 소품은 거기에 기여하지 않는다. 배경을 비우면 소싱·모델링 비용이 사라지고 스테이션 연출에 집중할 수 있다. 남이 만든 에셋 팩에 룩이 끌려가지도 않는다.
- **영향**: DESIGN 전면 교체. PLAN 2장(아트 방향)·4장(스테이션)·로드맵 갱신. 테마는 하늘 없이 조명 색·밝기만 바뀌고(저녁 노을 정도), 네온은 후순위로 내린다.

### 009. 모눈종이 바닥: 텍스처를 스크립트로 생성 + 앱과 코드 공유

- **날짜**: 2026-07-14
- **상태**: 채택
- **맥락**: 손그림 모눈종이 텍스처가 필요한데, CC0 3D 텍스처 사이트(ambientCG·Poly Haven)에는 실사 재질만 있고 손그림 노트 종이는 없다. 직접 그리거나(A), 코드로 생성하거나(B), 무료 이미지를 받아 쓰는(C) 선택지가 있었다. 셰이더로 모눈을 직접 그리는 안은 텍스처를 쓰지 않아 폐기했다.
- **결정**: (B) **텍스처를 코드로 생성해 PNG로 굽는다.** 생성 로직과 기본값은 `src/lib/gridPaper/`에 두고, **앱과 스크립트가 같은 코드를 쓴다.** `scripts/generate-grid-paper.mjs`가 그 모듈로 PNG를 굽고(1024px), 개발용 leva HUD(`GridPaperHUD`)가 같은 모듈로 브라우저에서 즉석 미리보기를 굽는다.
- **심리스 보장**: 바닥이 타일을 수십 번 반복하므로 이음매가 없어야 한다. 선의 떨림·농담은 **타일 폭을 정수 주기로 갖는 사인파**로만 만들고, 종이 결 노이즈는 격자 인덱스를 타일 폭으로 감아 반대편과 같은 난수를 참조하게 한다.
- **칸 크기 비율 기반**: 선 굵기·떨림 폭·떨림 주기를 픽셀이 아니라 **칸 크기에 대한 비율**로 정의한다. 칸 수(`cells`)만 바꿔도 모양이 유지된다.
- **이유**: 사람이 그린 손맛에는 못 미치지만 심리스가 보장되고, 값을 즉시 조절해 눈으로 맞출 수 있다. 나중에 직접 그린 이미지로 교체해도 배선은 그대로다(PNG만 갈아끼우면 된다).
- **영향**: leva를 개발 보조 도구로 도입. 프로덕션 번들에는 들어가지 않는다(dev 전용 렌더라 트리셰이킹됨).

### 010. 바닥은 조명을 받지 않는 재질(basic)

- **날짜**: 2026-07-14
- **상태**: 채택 (임시 — 테마 구현 시 재검토)
- **맥락**: 바닥을 `MeshStandardMaterial`로 두면 화면에 찍히는 밝기가 `종이색 × 광량`이 된다. three는 물리 기반 광량이라 diffuse 반사에 1/π가 걸리고, 지금 조명값에서 광량이 1에 못 미쳐 **흰 종이가 회색(`#D4CFC8`)으로 찍혔다.** 조명을 올리면 흰색이 되지만 씬의 모든 오브젝트가 함께 밝아진다.
- **결정**: 바닥만 **조명을 받지 않는 재질**(`MeshBasicMaterial`, `toneMapped: false`)로 둔다. 지정한 종이색이 화면에 그대로 나온다. 낮/밤은 조명이 아니라 **종이색에 색을 곱해** 표현한다.
- **이유**: 종이는 흰색이어야 하고, 그것을 위해 씬 전체 조명을 왜곡할 수는 없다. 톤 매핑(ACES)도 흰색을 회색 쪽으로 눌러 이 바닥만 제외한다.
- **한계**: 바닥과 나머지 오브젝트의 밝기 체계가 이원화된다. 테마(낮/저녁/밤)를 제대로 구현할 때 다시 본다.

### 012. 콘텐츠 스키마 확정: 컬렉션별 최소 정보 원칙, 프로젝트는 Firestore 최소 필드 + 로컬 상세 확장

- **날짜**: 2026-07-16
- **상태**: 채택
- **맥락**: Phase 7에서 콘텐츠 7종(profile·skills·experiences·education·awards·spec·projects)의 실제 필드를 정하며 데이터를 입력했다. PLAN.md 초안에는 `projects` 문서 예시로 `{ title, summary, tech[], images[], links, demoKey }`가 있었으나, 이 Firebase 프로젝트는 다른 프로젝트와도 공유하는 DB라 프로젝트마다 성격이 다른 상세 필드(기술스택·이미지·데모 연결)를 여기 넣기엔 부담이 있었다.

- **이유**: Firebase 프로젝트를 여러 프로젝트가 공유하므로, 이 포트폴리오에만 필요한 상세 데이터를 Firestore 스키마에 얹으면 다른 프로젝트와의 경계가 흐려진다. 최소 정보 + 로컬 확장으로 분리하면 Firestore는 가볍게 유지되고, 상세 UI 요구사항이 정해지는 시점(Phase 8)에 맞춰 로컬 데이터 구조를 유연하게 설계할 수 있다.
- **영향**: PLAN.md 6장 프로젝트 예시 갱신. FIRESTORE.md에 컬렉션별 확정 필드 반영. Phase 8에서 로컬 프로젝트 상세 데이터 구조를 새로 설계해야 한다.

### 014. 스테이션 영역: 클릭 판정 범위 = 근접 판정 기준, 거리 계산은 스테이션에 위임

- **날짜**: 2026-07-22
- **상태**: 채택
- **맥락**: 공통층은 스테이션을 점(배치 좌표)으로만 알고, 근접은 그 점까지의 거리로, 클릭은 임시 박스 메시로 판정했다. 그런데 실제 스테이션은 종이 위에 **면적을 차지하는 그림**이다. 페이지 한가운데에서만 반응하고 가장자리에서는 반응하지 않으면 클릭도 근접도 어긋난다.
- **결정**: 스테이션마다 종이 위에 차지하는 **영역**을 두고, 그 영역이 **클릭 판정 범위이자 근접 판정 기준**이 된다. 근접은 중심점이 아니라 **영역 테두리까지의 최단거리**로 재고(영역 안이면 0), 그 계산은 형태를 아는 스테이션이 레지스트리에 `distanceTo`로 등록한다. 등록하지 않은 스테이션은 공통층이 기존대로 배치 좌표까지의 거리로 잰다.
- **클릭 처리**: 영역 전체를 덮는 투명한 판에 스테이션 id를 실어 둔다. 포인터 핸들러가 없는 메시는 R3F 이벤트 대상이 아니므로 **우클릭 이동은 그대로 바닥으로 통과**하고, 좌클릭 활성화만 `Stations`의 레이캐스트가 잡는다.
- **이유**: 스테이션마다 모양이 완전히 다르다는 원칙(006·007)을 판정에도 적용한 것이다. 공통층이 원형 반경을 강요하면 사각형 페이지든 불규칙한 오브젝트든 전부 어긋난다. 반대로 도형 종류를 데이터로 열거하면(사각형·원·…) 종류가 늘 때마다 공통층을 고쳐야 하므로, 값이 아니라 **함수**를 맡긴다.
- **영향**: `StationEntry`에 `Inactive`·`distanceTo` 추가. 영역을 등록한 스테이션에서는 `NEAR_RADIUS`가 "중심점에서"가 아니라 "영역 테두리에서 바깥으로"를 뜻하게 된다. 이 영역이 닫기 조건(근접 이탈)에도 그대로 쓰인다.

### 015. Intro 스테이션: 첫 화면을 활성 상태로 시작하고, 활성화는 카메라 각도만 바꾼다

- **날짜**: 2026-07-22
- **상태**: 채택
- **맥락**: Intro는 이름·소개·사진처럼 **읽는 내용**이 전부라 3D로 일어설 오브젝트가 없다. 또 사이트에 들어오자마자 보여야 하는 첫인상이기도 하다.
- **결정**: 내용을 **종이(바닥)에 그려 둔 채로 두고**, 활성화하면 오브젝트를 건드리지 않고 **카메라만 정면(수직)으로 돌린다.** 정면에서 보면 웹페이지처럼 읽히고, 닫으면 항공뷰로 돌아가 다시 3D 맵이 된다. 그리고 **사이트 첫 화면이 이 활성 상태**다 — 진입 애니메이션 없이 처음부터 정면뷰로 시작한다.
- **배율은 건드리지 않는다**: zoom을 바꾸면 정면에서 보이는 크기가 실제 크기와 달라져 영역 크기를 판단할 수 없다. 각도만 바꾼다.
- **화면 방향**: 수직으로 내려다보면 기본 up(+y)이 시선과 겹쳐 화면 방향이 정해지지 않는다. up을 **월드 -z로 고정**해 영역의 가로·세로가 화면의 가로·세로와 맞게 한다. 바닥에 눕힌 글씨가 바로 읽히는 방향이기도 하다.
- **복귀 자세는 계산한다**: 돌아갈 항공뷰 자세를 "지금 카메라"에서 캡처하면 안 된다. 첫 화면은 팔로우가 한 번도 돌지 않아 카메라가 캐릭터를 바라본 적조차 없고, 다시 마운트되면 이미 정면뷰인 자세를 항공뷰로 잡는다. 팔로우 규칙(`캐릭터 위치 + followOffset`, 캐릭터를 바라봄)으로 **매번 계산**한다. 걸어나가는 중에도 종료 애니메이션이 돌기 때문에 매 프레임 다시 계산해야 마지막에 튀지 않는다.
- **이유**: "종이 위의 그림이 3D로 살아난다"가 시그니처지만, 모든 스테이션이 같은 방식일 필요는 없다(007). Intro는 읽는 내용이라 각도 전환만으로 2D↔3D의 인상 차이를 만드는 편이 내용에 맞고, 첫 화면을 활성으로 두면 방문자가 조작을 배우기 전에 소개부터 읽게 된다.
- **영향**: `useStationStore` 초기값이 `about-intro`/`active`. `useCameraStore`에 `followOffset` 추가. 첫 진입에 캐릭터를 숨겼다 등장시키는 연출은 실제 캐릭터를 만드는 Phase 11로 미룬다.

### 016. 크레파스 획을 코드로 생성

- **날짜**: 2026-07-22
- **상태**: 채택
- **맥락**: 나가기 화살표를 벡터 선으로 그렸더니 자로 잰 도형처럼 보여 스케치북 톤과 맞지 않았다. 손그림 느낌의 핵심은 매끈함이 아니라 **재질감**이다.
- **결정**: 크레파스 획을 **캔버스에 코드로 그려** 텍스처로 쓴다(`src/lib/crayon/`). 크레파스는 선이 아니라 **종이 결에 걸린 왁스 알갱이 더미**라는 관점으로, 경로를 촘촘히 훑으며 굵기 안쪽에 알갱이를 흩뿌린다. 가장자리로 갈수록 찍힐 확률을 낮춰 테두리를 너덜하게 만들고, 진하기를 저주파로 흔들어 눌린 구간과 뜬 구간을 만들고, 별도 마스크로 왁스가 끊기는 구간을 둔다. 경로 자체에도 미세한 떨림을 준다.
- **씨앗 기반 난수**: 렌더할 때마다 모양이 바뀌면 안 되므로 난수에 씨앗을 준다. 획마다 씨앗을 달리해 서로 다른 손놀림처럼 보이게 한다.
- **둥근 마무리**: 끝점 바깥으로 반지름만큼 더 나가되 폭을 원의 단면으로 줄여, 잘린 단면이 아니라 반원으로 맺는다.
- **이유**: 모눈종이 텍스처를 코드로 생성한 것(009)과 같은 계열이다. 이미지 에셋을 쓰면 색·굵기를 코드로 바꿀 수 없고, 스테이션마다 손그림 요소가 계속 필요한데 그때마다 그림을 그려 넣을 수는 없다. 유틸로 두면 경로만 주면 된다.
- **영향**: `lib/crayon` 추가. 나가기 화살표가 첫 사용처이고, 앞으로 다른 스테이션의 손그림 요소도 여기서 그린다.

### 013. 로드맵 재구성: 스테이션 관련 Phase(8~11)를 하나로 통합, 스테이션 단위 반복 진행

- **날짜**: 2026-07-16
- **상태**: 채택
- **맥락**: 기존 로드맵은 스테이션 디자인/모델링(8)·프로젝트 체험 데모(9)·출력 인쇄/PDF(10)·방명록(11)을 별도 Phase로 두고, 각 Phase가 전체 스테이션(또는 전체 콘텐츠)을 한 번에 처리하는 구조였다. 하지만 DECISIONS 006·007에서 이미 스테이션은 서로 독립적으로("공통 셸 없음") 구현하기로 정해뒀는데, Phase 구조는 스테이션 하나가 완성되기까지 여러 Phase(8→9→10→11)를 오가야 해서 이 원칙과 실행 순서가 어긋나 있었다.
- **결정**: 8~11을 하나의 Phase 8 "스테이션 상세 구현"으로 통합한다. 스테이션(About 3·Projects 3·Guestbook 1) 하나씩 **8-1 정적 오브젝트**(겉모습·배치 좌표) → **8-2 인터랙션 UI**(활성화 연출·Firestore 데이터 렌더링·카메라 제어, 프로젝트는 체험 데모, Guestbook은 쓰기 폼·봇 방지·보안 규칙 포함) → **8-3 인쇄 레이아웃**(PrintView 반영)을 완전히 끝내고 다음 스테이션으로 넘어간다. 진행 상황은 PROGRESS.md의 스테이션별 체크리스트(표)로 추적한다. 이후 Phase 번호(폴리싱·캐릭터+페인팅)는 9·10으로 재배치한다.
- **이유**: 스테이션 독립 구현 원칙과 실행 순서를 일치시키면, 한 스테이션을 처음부터 끝까지(겉모습→상세→인쇄) 완성한 뒤 다음으로 넘어가는 흐름이 더 명확해지고, 스테이션 단위 진행 상황도 한눈에 파악할 수 있다.
